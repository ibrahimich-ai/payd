# Payd — правила работы с проектом

## О проекте
Платформа Payd — система рассрочки с рабочими местами для разных ролей.
Стек: HTML + JavaScript (vanilla) + Supabase
Язык интерфейса: русский

## ГЛАВНЫЕ ПРАВИЛА (соблюдать всегда)

### 1. Никаких демо-данных
- Не создавать новые моки, хардкоды, плейсхолдеры, Lorem ipsum
- `data.js` — устаревший, постепенно переводим всё на Supabase
- При правке любого файла — если видишь обращение к `data.js`, 
  предложи замену на запрос к Supabase
- Пустые поля показывать как «—», а не заполнять фейками
- Вся работа с БД через `db.js` — не делать прямые вызовы Supabase из HTML-страниц

### 2. Справочники в БД
- Все списки (товары, тарифы, статусы, роли, способы оплаты, 
  причины отказа, шаблоны документов и SMS) — в таблицах Supabase
- Если справочника нет — сначала создай таблицу + seed с реальными 
  значениями + UI редактирования. Только потом подключай к карточке
- Изменение справочника НЕ должно ломать существующие заявки 
  (используем snapshot-поля)

### 3. Snapshot-правило
В заявках, договорах, графиках платежей хранить замороженные копии 
полей из справочников: snapshot_price, snapshot_markup_percent, 
snapshot_duration_months и т.д.

### 4. История и аудит
- Каждое изменение поля → в `audit_log` (заявка_id, поле, 
  старое значение, новое, user_id, timestamp)
- Каждый переход этапа сделки → в `application_stage_history`
- В карточке заявки — вкладка «История» с таймлайном

### 5. Бизнес-цепочка заявки
Создание (партнёр) → Проверка СБ → Приём первого взноса (кассир) 
→ Договор → Подписание → Выдача

Каждый этап — отдельное рабочее место со своей очередью и правами.
На каждом переходе — запись в историю и уведомление следующей роли.

### 6. Аутентификация и роли
- Только Supabase Auth, никакого фейкового логина
- Роль определяется через таблицу `user_roles`, не через 
  email или localStorage
- RLS включён на всех таблицах с пользовательскими данными
- Редирект после логина — по роли (партнёр → partner.html, 
  кассир → kassir.html и т.д.)

### 7. Безопасность данных
- Никаких секретов в коде, только в .env
- Никогда не логировать паспорта, пароли, токены
- Для обращения к товару/клиенту — UUID, а не предсказуемые ID

## Процесс работы

- **Перед изменениями** — показывай план и жди подтверждения
- **Перед миграцией БД** — показывай SQL и объясни, что меняется
- **Не удаляй данные** без явного подтверждения
- **Большие задачи** — разбивай на этапы, отчитывайся после каждого
- **Если что-то неоднозначно** — задай вопрос, не угадывай
- **Каждое изменение в коде** — короткое объяснение «что и зачем»

## Чего НЕ делать

- Не создавать новые `mockData.js` / `fakeData.js` / похожие
- Не использовать `console.log` вместо реальной записи в БД
- Не оставлять `// TODO` без объяснения, что именно надо доделать
- Не менять схему БД без миграции
- Не коммитить .env и ключи Supabase
- Не удалять `data.js` одним махом — только постепенная деградация 
  с заменой каждого использования на Supabase
