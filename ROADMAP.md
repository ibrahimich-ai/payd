3. Проверить, есть ли уже записи в `application_history` со «сломанным» `app_id` — если да, миграция: удалить или попытаться сматчить по `number`.
4. Проверить, не создаёт ли retry-механизм outbox дубли в `applications` (а не только в `application_history`) — это и есть гипотеза, почему `#8270156221` лёг дважды.
5. После фикса — наблюдать журнал на предмет повторных дублей.

**Зачем:** дубли заявок — критическая проблема (видно клиенту, ломает счётчики KPI, порождает лишние записи в `payments`).

### Не-uuid id-генераторы для uuid-PK таблиц (тот же класс бага, не выстрелил)

**Контекст.** При фиксе бага audit log uuid (см. раздел выше) выяснилось: помимо `applications`, в проекте ещё несколько uuid-PK таблиц, для которых id генерируется в формате `<префикс>_<Date.now>`. По коду db.js:608 список uuid-PK таблиц: `partners, applications, cash_ops, payouts, reserv_stages`. Из них в коде создаются:

- `cash_ops` — [zayavka.html:2157, 2247, 2364](zayavka.html#L2157), [kassa.html:641](kassa.html#L641): `'op_' + Date.now()`
- (и через `applications` — уже починено)

Также есть некритичные (не в списке uuid-PK, но всё равно стоит унифицировать на `PaydDB.utils.uuid()`):

- `cashes` — [kassa.html:483](kassa.html#L483): `'c_' + Date.now()`
- `cash_articles` — [kassa.html:850](kassa.html#L850): `'a_' + Date.now()`
- `tariffs` — [settings.html:1177](settings.html#L1177): `'t_' + Date.now()`
- `product_categories` — [products.html:333](products.html#L333): `'cat_' + Math.random()`

**Что произойдёт, когда выстрелит** (для uuid-PK):
1. Запись в localStorage с `'op_<num>'` id.
2. `pushCollection` пушит в облако, нормализатор удаляет non-uuid id (теперь с warn — после фикса audit-бага).
3. Сервер генерирует свой uuid.
4. Realtime эхо приходит с серверным uuid → handleRealtimeChange ищет по pk, не находит → дубль в localStorage.

То же самое, что было с `applications` 2026-05-08, просто пока не выстрелило, потому что cash_ops через realtime реже эхает на тех же страницах.

**Что сделать:**
1. Заменить генераторы на `window.PaydDB?.utils?.uuid?.()` с фолбэком на старый формат (тот же паттерн, что в applications-фиксе).
2. Расширить `cleanupBrokenAppIds` в db.js до универсальной `cleanupBrokenIds(coll)`, прогонять для всех uuid-PK таблиц после `backgroundPull`.
3. Аналогично — расширить outbox guard, если для других таблиц появится audit-логирование.

**Зачем** (приоритет):
- Не выстрелило — но это вопрос времени. cash_ops пишутся при каждом платеже, и если кто-то откроет zayavka.html (платежи) на двух устройствах одновременно — возможен дубль платежа.
- Платёж задвоить **опаснее**, чем заявку: ломает баланс кассы, остаток долга, отчёты по выручке.

**Зависимости:** требует уже применённого фикса audit log uuid (для `PaydDB.utils.uuid()`).

**Контекст в истории:** разговор от 2026-05-09 о фиксе бага audit log uuid — там же диагностика причины (рассинхрон id между localStorage и облаком через `Storage.prototype.setItem` перехватчик + realtime).

### Разделение концепций: Тип сделки vs Тариф

**Задача.** В упрощённой форме `zayavka-new.html` два выпадающих списка («Тип сделки» и «Тариф») концептуально путаются с тем, что показывается в карточке заявки `zayavka.html` — там в поле «Тип сделки» смешаны и типы (Товар со склада, Товар от партнёра, Под заказ, Услуга), и тарифы (Акция 0% на 6 мес, ВИП, Базовый). Это две независимые сущности, которые нужно разделить.

**ТИП СДЕЛКИ** (`deal_types`):
- Что продаём: товар со склада, товар от партнёра, услуга, под заказ.
- Влияет на: участников цепочки (нужен ли партнёр), формирование складских движений, генерацию документов.

**ТАРИФ** (`tariffs`):
- На каких условиях продаём: ставка наценки, срок, минимальный взнос, льготные условия.
- Влияет на: формулу расчёта суммы рассрочки.

**Связь.** Одна заявка имеет один тип И один тариф. Они независимы: один и тот же товар можно продавать по разным тарифам, одна акция применима к разным типам товаров.

**Известные баги (видны до разделения):**
- При выборе тарифа «Акция 0%» расчёт показывает наценку 112 000 ₽ при ставке 0,00% (итого 111 998 ₽ при цене 112 000 ₽).
- Тариф «Акция 0%» выбирается по умолчанию — партнёры могут не заметить и продать по акции случайно.

**Что сделать:**
1. SQL-миграции для таблиц `tariffs` и `deal_types` (с seed реальных значений).
2. UI: разделить два выпадающих списка везде, где они встречаются (`zayavka-new.html`, `zayavka.html`, фильтры в `zhurnal.html`).
3. Логика расчёта: формула берётся из `tariffs`, а не из типа сделки.
4. Тариф по умолчанию — «Базовый» (или вообще не подставлять, сделать обязательным выбор).
5. Snapshot и тип, и тариф в `applications` (правило #3 CLAUDE.md).

**Зависимости:** фильтры по типу в `zhurnal.html` могут перестать работать после разделения — пересобрать.

### 🎯 Единая логика расчёта — Тариф как единый источник правды

**Контекст.** Сейчас формула расчёта рассрочки (наценка, срок, мин. взнос, доступные продукты) дублируется как минимум в 4 местах: мини-калькулятор (`app.js openMiniCalc`), полный калькулятор (`calculator.html`), форма создания заявки (`zayavka-new.html`), карточка заявки (`zayavka.html`). Каждое место считает по-своему, ставки захардкожены, списки сроков и продуктов разъезжаются. Последние коммиты (`21a057b`, `64594bb`) починили только синхронизацию mini ↔ full, но не источник правды.

**Цель.** Тариф из `settings.html` → `Тарифы и условия` становится **единственным источником правды** для всех расчётов в системе. Никаких захардкоженных ставок, сроков, мин. взносов и списков продуктов в калькуляторах и формах.

#### Объект тарифа (структура в БД)

Таблица `tariffs`:

| Поле | Тип | Описание |
|---|---|---|
| `id` | uuid | PK |
| `name` | text | «Базовый», «ВИП», «Акция 0%» |
| `description` | text | пояснение для UI |
| `is_default` | boolean | один тариф = по умолчанию |
| `is_active` | boolean | можно отключить без удаления |
| `products` | jsonb | `{ standard: bool, murabaha: bool, ijara: bool }` — какие продукты включены |
| `terms` | int[] | например `[3, 6, 12, 18, 24, 36]` — какие сроки доступны клиенту |
| `min_down_percent` | numeric | мин. первый взнос в % |
| `max_amount` | numeric | максимальная сумма заявки |
| `rates` | jsonb | `{ murabaha: { 3: 15, 6: 20, 12: 30, ... }, ijara: { ... }, standard: { ... } }` — базовые ставки наценки по сроку |
| `formula` | jsonb | конфиг формулы зависимости ставки от первого взноса (см. ниже) |
| `created_at` / `updated_at` | timestamptz | |

Конфиг `formula` — обсуждается завтра, варианты:
- `{ kind: 'linear', baseMultiplier: 0.5 }` → `rate = baseRate × (1 - downPercent/100 × 0.5)`
- `{ kind: 'inverse', factor: 0.5 }` → `rate = baseRate + baseRate × 0.5 / (downPercent / 100)`
- иное — согласовать с пользователем (см. вопрос №1 ниже).

#### Главное правило

Тариф с `is_default=true` — **единственный источник правды** для всех расчётов в системе. Все потребители читают из БД, не считают и не предполагают сами.

#### Потребители тарифа (читают из БД)

**1. Мини-калькулятор** (`app.js → openMiniCalc`):
- Кнопки продуктов («Стандартная / Мурабаха / Иджара») рендерятся **только** для тех, что в `tariff.products[*] === true`.
- Кнопки сроков рендерятся **только** для значений из `tariff.terms`.
- Ставка по сроку — из `tariff.rates[product][term]`.
- Минимальный взнос — `tariff.min_down_percent`.

**2. Полный калькулятор** (`calculator.html`):
- То же самое, что мини.
- Дополнительно — детальный график платежей.

**3. Форма создания заявки** (`zayavka-new.html`):
- В «Тип сделки» — список продуктов **только** включённых в выбранном тарифе.
- В «Срок рассрочки» — **только** значения из `tariff.terms`.
- В «Тариф» — **все активные** тарифы из `tariffs WHERE is_active=true`, по умолчанию `is_default=true`.
- Ставка наценки — `tariff.rates[product][term]` × формула с учётом первого взноса.
- Мин. взнос — `tariff.min_down_percent`.

**4. Карточка заявки** (`zayavka.html`):
- Тариф **не редактируется** — снапшот зафиксирован при создании заявки.
- Только показывает что было выбрано и считает по снапшоту, а не по текущему тарифу.

#### Текущие баги (примеры рассинхрона)

Эталон — тариф «Базовый» (по умолчанию):
- Включена **только** Мурабаха (Стандартная и Иджара выключены).
- Сроки: 3 / 6 / 12 / 18 / 24 / 36 мес.
- Мин. взнос: 20%.
- Макс. сумма: 3 000 000 ₽.
- Ставки Мурабаха: 15% / 20% / 30% / 16% / 20% / 28% по сроку 3 / 6 / 12 / 18 / 24 / 36.

**Что не так в калькуляторе** (`calculator.html` + мини):
- Показаны кнопки «Стандартная» и «Мурабаха» — должна быть **только** Мурабаха.
- Кнопки сроков «12 / 24 / 36 / 48» — но 48 не должно быть, а 6 и 18 — должны.
- Ставки в калькуляторе свои, не из настроек тарифа.

**Что не так в форме создания заявки** (`zayavka-new.html`):
- В выпадающем «Тариф» перечислены: «Акция 0%», «ВИП», «Базовый», что-то ещё — смесь продуктов и тарифов (см. также раздел «Разделение концепций» выше).
- Расчёт наценки фиксированный (`20% × срок`), не зависит от размера первого взноса.
- В «Тип сделки» перечислены продукты, не настроенные в активном тарифе.

#### План реализации (по шагам)

**Шаг 1. Структура и сохранение тарифов в БД** (~2 ч)
- Миграция: создать таблицу `tariffs` с полями выше.

#### Вопросы к обсуждению с пользователем

1. **Точная формула** «больше первый взнос → меньше наценка»?
   - Вариант A: `rate = baseRate × (1 - downPercent/100 × 0.5)` (линейная скидка).
   - Вариант B: `rate = baseRate + baseRate × 0.5 / (downPercent / 100)` (обратная зависимость).
   - Иное — пользователь утверждает на старте Шага 2, опираясь на текущую логику в `calculator.html` после коммитов `21a057b` / `64594bb`.

2. **Старые заявки в БД** — пересчитывать или нет?
   - Гипотеза: **не пересчитывать.** У них уже есть зафиксированный тариф/тип/наценка, снапшот сделан при создании. Это согласуется с правилом #3 CLAUDE.md.

3. **Новый тариф через кнопку «+ Новый тариф» в `settings.html`** — становится `is_default` или нет?
   - Гипотеза: **нет.** При добавлении `is_default` остаётся у старого тарифа; перевести флаг можно только явным действием в UI («Сделать тарифом по умолчанию»). Иначе случайное добавление меняет правила для всей системы.

#### Подзадачи (из работы 2026-05-09 над `PaydDB.calc.murabahaPayment`)

- **TYPES в mini-calc → динамика из `tariff.types`.** Сейчас в [app.js:1320](app.js#L1320) словарь `TYPES = { standard, murabaha, ijara }` хардкодится. После единой `calc.deal()` имеет смысл вынести лейблы (Стандартная/Мурабаха/Иджара) в `PaydDB.tariffs.LABELS` или загружать из самого тарифа (если у каждого продукта будет свой `displayName`). По образцу `TYPE_LABELS` в [calculator.html](calculator.html).
- **Disabled-state для кнопок сроков без ставки.** Сейчас в [calculator.html:1050-1052](calculator.html#L1050-L1052) и [app.js:1420-1421](app.js#L1420-L1421) сроки без `murabaha_rates[term]` **скрываются** через `filter`. UX-улучшение: показывать disabled с подсказкой «Срок недоступен для Мурабахи». Не блокер (фильтрация защищает от 12%-магии), но помогает пользователю понять конфигурацию тарифа.
- **Расхождение семантики `murabaha_rates` между калькуляторами и формой заявки.** В [calculator.html](calculator.html) и [app.js](app.js) Мурабаха-формула: `markup = body × rate/100 × (term/12)` — `rate` трактуется как **годовая ставка для клиента**. В [zayavka-new.html:650-664](zayavka-new.html#L650-L664) формула: `markup = base × rate/100` — `rate` трактуется как **разовая наценка к закупочной цене партнёра**. Это разная бизнес-семантика на одинаково именованном поле, что путает при настройке тарифа в settings. Решение в рамках единой `calc.deal()`: либо два разных поля (`client_apr` и `partner_markup_pct`), либо явное обозначение в `tariff.calc_mode`. Требует продуктового решения «что именно настраивает админ в редакторе тарифа».
- **z-new не подставляет is_default тариф автоматически.** [zayavka-new.html:786-793](zayavka-new.html#L786-L793) при отсутствии валидного драфта `tariffId` сбрасывает в пустую строку, требуя ручного выбора. После Шага 1A (is_default в БД) логичнее автоподстановка дефолта.
- **calculator.html — standalone-страница вне db.js по design'у.** Только inline `<script>`, никаких внешних зависимостей. Шаги 2-4 текущей итерации для неё де-факто fallback-only: `window.PaydDB` всегда undefined, optional chain `PaydDB?.calc?.murabahaPayment` всегда падает на локальный fallback. Формула fallback и логика `loadGlobalTariffs()` тождественны единой версии в [db.js](db.js) (проверено замерами на 3 кейсах). Не блокер — работает корректно. Архитектурно странно: будущему разработчику может быть неочевидно, что вызовы `PaydDB.*` в calculator.html — декоративные. Зафиксировано в comments каждого callsite (см. коммит после `51d8b8b`).
- **Partner-priority в standalone calculator.html.** Сейчас на public-странице `partner.tariff` из `payd.partners.v1` не учитывается (db.js не подключён → `tariffsGetActive` недоступен → fallback на `loadGlobalTariffs` без partner-priority). Не критично: партнёр приходит через `?partner=<name>` URL и может явно указать тариф через `?tariff=<id>`. Если понадобится — рассмотреть локальный `resolvePartnerInline()` в самом calculator.html (повторение части логики `getActive` без вызова PaydDB), не подключая db.js.
- **`tariffs.created_at = NULL` для t_basic** — наследие insert'а до добавления `default now()` в схеме (выявлено при миграции is_default 2026-05-10). Одна команда: `UPDATE tariffs SET created_at = now() WHERE created_at IS NULL`. Не блокер: `is_default` срабатывает приоритетом 2 в `tariffsGetActive` до фолбэка по `created_at`. Cleanup стоит сделать при следующей миграции тарифов.
